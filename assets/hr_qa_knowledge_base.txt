# PROJECTS

Q: Explain your Digital Twin project.
A:
So the idea was — what if there was an AI version of me that could answer questions about my background, my projects, my skills, the way I actually would? That's the Digital Twin. I built it using an Agentic RAG architecture, which means it doesn't just retrieve and answer — it actually reasons about what tool to use. If you ask it something about me, it searches my files. If you ask something general, it goes to the web. I used LangChain for the agent logic, ChromaDB for the vector database, and Streamlit for the UI. The whole thing runs on GPT-4o-mini. It was genuinely one of the most fun projects I've built because I basically got to design an AI that thinks like me — or at least tries to.

Q: Explain your multi-modal pipeline.
A:
Real-world data is messy — it's not always clean text. So I built a pipeline that can handle PDFs, code files, CSVs, and plain text all in one system. Each file type goes through its own loader — PyPDFLoader for PDFs, TextLoader for code and markdown, CSVLoader for structured data — gets converted into embeddings, stored in a vector DB, and then retrieved when needed. It makes the system actually useful beyond toy examples, because it can handle whatever you throw at it.

Q: How does your agent decide what to do?
A:
It uses a tool-calling mechanism, so it's not just blindly retrieving every time. The agent looks at the question, figures out what kind of answer is needed, and then picks the right tool. If you're asking about me — my projects, my background — it hits the RAG tool and searches my files. If it's something general or current, it goes to DuckDuckGo. The prompt guides the decision, but it's the reasoning layer that makes it feel like it's actually thinking rather than just pattern matching.

Q: Difference between RAG and an agent?
A:
RAG is more like a smart search — you ask something, it finds relevant chunks, and the LLM uses that to answer. It's passive. An agent is active — it decides what to do, which tool to use, and can take multiple steps before giving you an answer. RAG is a tool that an agent can use. So in my Digital Twin, RAG is one of the tools the agent has access to, not the whole system.

A:
Step by step, always. Guessing is the worst approach — it just wastes time and creates more confusion. I check the logs first, isolate which module is failing, test that part in isolation, and fix incrementally. There's something almost satisfying about narrowing it down to one line that broke everything. Almost.

Q: Tell me about a challenge you faced and how you solved it.
A:
Deploying my Digital Twin on Streamlit Cloud was honestly a whole adventure. ChromaDB kept throwing a SQLite error because the cloud environment is read-only. I went through five different errors — grpcio build failures, protobuf conflicts, missing ddgs package — one after another. But I fixed each one systematically, reading the tracebacks carefully and researching each dependency issue. Eventually switched to EphemeralClient for ChromaDB which runs fully in-memory and that solved it. It taught me a lot about how deployment environments differ from local dev.
